WordPress 換 CMS 的痛點「自行搬遷」實際上幾乎不可行。

WordPress 的匯出機制是為「同樣是 WP 站」設計，換到其他 CMS 即會出現嚴重相容性問題：Page Builder 內容變成 HTML 雜訊、圖片連結失效、URL 變動、SEO 流量大幅流失。中小企業沒有人力處理這些。

WP 官方匯出 XML 結構混亂

官方匯出格式是為 WP 自身設計：圖片使用絕對網址、page builder 的 shortcode 內嵌、自訂欄位混雜。換到其他 CMS 等於要全部重新解析。

典型解析失敗率 30–60%（視 builder 種類而定）

100 篇文章需逐篇複製貼上

無專用工具時只能手動處理：標題、內文、圖片、SEO 描述、發佈日期、分類等，每篇 5–10 分鐘。100 篇 = 一人滿班 2–3 週。

典型耗時 8–25 工作天（依內容複雜度）

URL 一變動 SEO 流量大幅流失

新站若使用新 slug，Google 既有排名等於歸零重來。未做 301 重導對照表時，原本第一頁的關鍵字會掉到 5–10 名外，需要 3 個月才能恢復。

流量典型下降 40–70%（未做 301 重導時）

搬完還需要重排版並補 schema

即使搬遷完成，舊站內容缺乏結構，AI 引擎無法解析。仍需重新標 schema、補 FAQ、產 llms.txt，等於從 WP 撈出後還要再做一次 GEO 原生化工程。

後續再 1–2 週進行內容結構化

AICMS 的解法WordPress 專用爬蟲與 AI 重排，一條龍服務。

本工具非通用 HTML scraper，而是專為 WordPress 結構設計。優先呼叫 REST API 取得乾淨資料，失敗時再 fallback 至 HTML 解析。各家 Page Builder 規則皆內建。

REST API 優先，HTML 為備援

WP 4.7 以上預設啟用 /wp-json/ REST API，可直接取得乾淨 JSON。未啟用時 fallback 走 sitemap.xml 與 HTML 雙重解析，兩條路徑互為備援。

REST · sitemap · OG · JSON-LD

Page Builder 解析

Elementor、Divi、Gutenberg、WPBakery、Beaver Builder 各家 page builder 的 shortcode 與 block 規則皆內建。不是搬遷整段 HTML，而是還原成乾淨的結構化內容。

5+ builders supported

合約權責不是直接爬取，而是先把權責定清楚。

網站遷移涉及智慧財產權、資料隱私、SEO 流量風險。我們堅持每個案件都簽署正式合約，雙方權責白紙黑字。這是對客戶與服務商雙方的保障，並非單純的法律程序。

合約核心條款

四件事，動工前先講清楚。

常見問題是「能否直接幫忙把網站搬過來？」答案是可以，但必須先簽合約，否則無法保障客戶與服務商雙方。

1

智慧財產權歸屬

客戶須為原網站內容所有人（或經授權代理人）。我們會要求出示網站擁有權證明（DNS 控制權與 WP 管理者帳號）。原站文字與圖片版權歸客戶，遷移後新站版權仍歸客戶。

條款引用：服務合約 §3 智財權條款
2

資料隱私（PDPA / GDPR）

若原站含會員資料、訂單、Email 訂閱清單，依個資法規範處理：傳輸全程加密、僅用於遷移用途、完成後 30 天內銷毀我方端的暫存。客戶可隨時要求查驗銷毀證明。

條款引用：服務合約 §5 個資保護條款
3

原站既存問題不轉嫁

遷移工程不負責修復原站既有問題：版權爭議圖片、過去的 SEO penalty、第三方 plugin 殘留後門等。掃描階段會列出已知風險，由客戶決定處理方式。

條款引用：服務合約 §7 既存風險揭露
4

透明流程與客戶簽核

遷移各階段皆有客戶簽核點（步驟 1、3、5）。客戶可隨時暫停、要求調整或終止。我們保留爬蟲動作日誌，客戶可隨時調閱查驗。明文禁止任何未告知的修改或上線。

條款引用：服務合約 §4 服務透明性
對客戶的承諾：合約範本可在線上會議中逐條檢視討論調整。若客戶法務有特殊規範（NDA、DPA 等），可依客戶版本簽署。未簽合約一律不開工。

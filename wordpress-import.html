WP 換 CMS 為什麼這麼痛「自己搬」幾乎不可能。

WordPress 本身的匯出機制就是給「同樣是 WP 站」用的，換到別的 CMS 立刻一團亂。Page Builder 內容變 HTML 雜訊、圖片連結壞、URL 換、SEO 全掉。中小企業根本沒人力處理。

📦

WP Export XML 一團亂

官方匯出格式是給 WP 自己用的。圖片用絕對網址、page builder 的 shortcode 嵌入、自訂 fields 混在一起。換到別的 CMS 等於要重新解析所有東西。

🔧 典型解析失敗率 30-60%（看 builder 種類）
⏰

100 篇要一篇一篇貼

沒有專用工具就只能複製貼上 — 標題、內文、圖片、SEO 描述、發佈日期、分類⋯每篇 5-10 分鐘。100 篇 = 一個人滿班搞 2-3 週。

⏱️ 典型耗時 8-25 工作天（看內容複雜度）
📉

URL 一換 SEO 全掉

新站如果用新 slug，Google 既有排名要從零再爬。沒做 301 對照表，原本第一頁的關鍵字會掉到 5-10 名外，3 個月才養得回來。

📊 流量典型掉 40-70%（沒做 301 重導向時）
🧩

搬完還要重排版 + 補 schema

就算搬完，舊站內容沒結構，AI 引擎看不懂。要重新標 schema、補 FAQ、做 llms.txt — 等於從 WP 撈出來後還要再做一次「GEO 原生化」工程。

🛠️ 後續再花 1-2 週 內容結構化

AICMS 怎麼做WordPress 專用爬蟲 + AI 重排，一條龍。

我們的爬蟲不是通用 HTML scraper，是「看得懂 WordPress 結構」的專用工具。先試 REST API 拿乾淨資料，失敗才 fallback 到 HTML 解析。Page Builder 各家規則都內建。

🔌

REST API 優先 + HTML fallback

WP 4.7+ 預設啟用的 /wp-json/ REST API 直接拿乾淨 JSON。沒開的話 fallback 走 sitemap.xml + HTML 雙重解析。兩條路徑相互備援。

REST · sitemap · OG · JSON-LD
🧱

Page Builder 解析

Elementor / Divi / Gutenberg / WPBakery / Beaver Builder — 各家 page builder 的 shortcode / block 規則內建。不是把整段 HTML 搬過去，而是還原成乾淨結構化內容。

5+ builders supported
🖼️

媒體自動下載 + CDN re-host

原站圖片 / PDF / 影片自動撈到 AICMS Storage (Supabase CDN)。連結重新指向新網域，原站關了不影響。retina / WebP 自動產出。

Supabase Storage · 自動 retry

URL slug 保留 + 301 對照表

原 WP 的 slug 在新站照搬，SEO 不掉。若必須改 slug，自動產出 301 重導向對照表 (CSV)，給 Cloudflare / 主機 redirect 使用。

slug map · 301 csv export

post-import AI · 自動 schema

掃 sitemap.xml 出完整頁面目錄、辨識 page builder 種類、列出媒體檔總量。產出「遷移評估報告」給客戶確認範圍跟工程量。

sitemap 掃描 builder 偵測 媒體總量盤點 評估報告 PDF

3

樣本爬取 + 客戶預覽

先爬 5-10 頁代表性內容（首頁、最熱門 3 篇、最複雜結構 1 篇），讓客戶用 /preview 連結看效果。確認版型對、結構對、媒體沒漏，才進下一步。

樣本爬取 /preview 連結 客戶簽核

合約權責不是隨便爬一爬，先把規矩定清楚。

網站遷移涉及智慧財產權、資料隱私、SEO 流量風險。我們堅持每個案件都簽正式合約，雙方權責白紙黑字。這不是法律包袱，是對客戶跟對我們自己的保護。

📋 合約核心條款

4 件事，動工前先講清楚。

客戶常問「能不能直接幫我把那個站搬過來？」— 答案永遠是：能，但要先簽合約。否則我們無法保護客戶、也無法保護我們自己。

1

智慧財產權歸屬

客戶須為原網站之內容所有人 (或經授權代理者)。我們會要求出示網站擁有權證明 (DNS 控制權 + WP 管理者帳號)。原站文字、圖片版權歸客戶；遷移後新站版權仍歸客戶。

📜 條款引用：服務合約 §3 智財權條款
2

資料隱私 (PDPA / GDPR)

若原站含會員資料、訂單、Email 訂閱清單，我們依個資法處理：傳輸全程加密、僅用於遷移用途、遷移完成後 30 天內銷毀我方端的暫存。客戶可隨時要求查驗銷毀證明。

📜 條款引用：服務合約 §5 個資保護條款
3

原站既存問題不轉嫁

遷移工程不負責修原站既有問題：版權有爭議的圖片、過去的 SEO penalty、第三方 plugin 留下的後門等。我們會在掃描階段列出已知風險，由客戶決定處理方式。

📜 條款引用：服務合約 §7 既存風險揭露
4

透明流程 + 客戶簽核

遷移每個階段都有客戶簽核點 (Step 1, 3, 5)。客戶可隨時暫停、要求調整或終止。我們保留每個爬蟲動作的日誌；客戶查驗時須能調閱。任何「偷偷加料」「悄悄上線」都明文禁止。

📜 條款引用：服務合約 §4 服務透明性
💡 我們對客戶的承諾：合約範本可在線上會議時直接 review，所有條款可逐條討論調整。若客戶現有法務有特殊規定 (NDA / DPA 等)，我們可依客戶版本簽署。沒簽合約一律不開工。
